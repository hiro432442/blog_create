# はじめに：その「1分」は、積み重なると「人生」になる

エンジニアの皆さん、今日一日で何回 `git status` を叩きましたか？何回 `cd` コマンドでディレクトリを移動しましたか？
「そんなこと、考えたこともない」。そう答えるのが普通でしょう。

しかし、たった1回3秒のコマンド入力も、1日100回やれば5分。1年で約20時間です。
チリも積もれば山となる……という話ではありません。

問題なのは「時間」ではなく「思考の中断（コンテキストスイッチ）」です。
「えーと、あのディレクトリのパスは何だっけ？」
「あのGitコマンドのオプションは何だっけ？」
この一瞬の迷いが、せっかく乗っていた「フロー状態（没頭）」を断ち切るのです。

この記事では、2025年のシニアエンジニアがこだわるべき、最強の「俺用DX（Personal Developer Experience）」構築術について語ります。

# 2025年の「俺用DX」：シニアがこだわる3つの領域

GoogleやAmazonが提供するCI/CDパイプラインだけがDXではありません。あなたの手元のシェル環境こそが、最も重要なDXの源泉です。

## 1. The "Terminal" as an Extension of Thought（思考速度のターミナル）

ターミナルは、あなたの思考をOSに伝えるための「神経接続端子」です。神経の伝達速度を上げましょう。

*   **Dotfiles**: 自分の環境設定（`.zshrc`, `.config/nvim` など）をGitで管理していないエンジニアは、パンツを穿かずに戦場に出るようなものです。GitHubで `dotfiles` リポジトリを作り、どのマシンでも `git clone` 一発で「いつもの環境」が再現できるようにしましょう。

*   **Modern Tools**: まさかまだ `ls` や `cat` を使っていませんか？
    *   `ls` → **`eza`**: カラーリングとアイコン表示で、ファイルの種類が一目でわかる。
    *   `cat` → **`bat`**: シンタックスハイライト付きでコードが読める。
    *   `cd` → **`zoxide`**: 過去の移動履歴を学習し、`z blog` だけで `~/projects/my-blog-site` に飛べる。
    これらはRust製のモダンツール群で、圧倒的な高速動作と視認性を誇ります。

*   **Alias**: 例えば `git commit -am "fix"` を毎回打つのは拷問です。`gcam "fix"` で済ませましょう。「打鍵数を減らす」ことは「脳の負荷を減らす」ことと同義です。

## 2. AI-Integreated Workflow（AIをCLIに住まわせる）

エラーが出た時、ブラウザを開いてStack OverflowやChatGPTに貼り付けていませんか？
その「ウィンドウの切り替え」すらもムダです。2025年は、ターミナルの中にAIを住まわせる時代です。

*   **GitHub Copilot CLI / Shell GPT**:
    「ポート3000を使っているプロセスをキルしたい」と思ったら、コマンドをググるのではなく、AIに聞くのです。
    `?? kill process on port 3000`
    これだけで、AIが適切なコマンド（`lsof -i :3000 | xargs kill`）を提案してくれます。

コミットメッセージも、`git diff` の内容をAIに読ませて自動生成させる。人間はそれを読んで「OK」と押すだけ。
ターミナルから一歩も出ずに開発を完結させる。それが最強の没入感を生みます。

## 3. Programmable Keyboards & Shortcuts（ハードウェアまでハックする）

キーボードショートカットは「与えられたものを覚える」ものではありません。「自分で作る」ものです。

*   **Raycast / Alfred**:
    Macユーザーなら、Spotlightの代わりにこれらを使いましょう。
    「Jiraのチケット作成」「特定のAPIドキュメントを開く」「カラーコードの変換」。これら全てをキーボードショートカット一発で呼び出せるようにスクリプトを書きます。

*   **Window Management**:
    ウィンドウをマウスでドラッグして配置するのは、石器時代の所作です。
    左半分にエディタ、右半分にブラウザ。これもショートカット一発でゼロ秒で配置する。
    究極の目標は、「こうしたい」と脳が思った瞬間に、画面がその状態になっていることです。

# 警鐘：自動化の「沼」に気をつけろ

ここで有名な [xkcdのグラフ](https://xkcd.com/1205/) を思い出してください。
「タスクを自動化するためにかける時間が、実際に節約できる時間を超えてはいけない」という戒めです。

私たちエンジニアは、設定ファイルをいじるのが大好きです。
「Vimのプラグイン選定で週末が潰れた」「シェルのプロンプトをカッコよくするのに3時間かけた」。
それはそれで楽しい趣味ですが、仕事ではありません。

**YAGNI（You Aren't Gonna Need It）**の原則は、コードだけでなく開発環境にも適用されます。
「いつか使うかもしれないエイリアス」は定義しない。本当に繰り返して苦痛を感じた作業だけを、スクリプトにする。
目的はあくまで「プロダクトをユーザーに素早く届けること」であり、「最強の開発環境を自慢すること」ではないことを忘れないでください。

# 結論：環境設定は「愛」だ

弘法は筆を選ばないと言いますが、一流のエンジニアは間違いなく筆を選び、そして自分で筆を作ります。

毎日8時間以上触れる道具を、自分好みに磨き上げること。それは効率化のためであると同時に、**「自分自身へのリスペクト」**であり、**「プロフェッショナリズムの現れ」**でもあります。

使いにくい道具で我慢して仕事をすることは、自分を大切にしていないのと同じです。
今日から、あなたがイライラしているその作業を、ひとつだけシェルスクリプトにしてみてください。
その小さなスクリプトが、あなたのエンジニア人生を少しだけ、しかし確実に豊かにしてくれるはずです。
