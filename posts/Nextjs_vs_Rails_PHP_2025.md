# 導入：フロントエンドの「重力」に疲れていませんか？

「たかが管理画面の入力フォームを作るだけなのに、なぜ`useState`だ`useEffect`だと騒がなければならないんだ？」
「PHPならファイルをアップロードして終わりだった仕事が、Next.jsだとビルドパイプラインの構築から始まる…」

正直に言いましょう。今のWeb開発は「複雑」です。
現場のエンジニアの中には、JavaScriptエコシステムの異常な早さの進化（JavaScript Fatigue）に、静かな疲れを感じている人も多いはずです。

「昔（RailsやPHPの時代）は良かった」と懐古するのは簡単です。しかし、ユーザーはもう「ページ遷移のたびに画面が白くなるWebサイト」を許容してくれません。リッチなUI/UXは、もはや「あったら良い」ものではなく「前提条件」になりました。

この記事では、現代のデファクトスタンダードである **Next.js (App Router)** と、長い歴史を持つ **Rails / Laravel (MVCフレームワーク)** を比較し、私たちが技術選定で「何を得て、何を失うのか」を冷静に分析します。

![Tech Stack Comparison](/Engineerblog/globe.svg)

# 構造的な5つの違い：メリットとデメリット

## 1. Rendering Strategy: RSC vs MVC
最大の違いは「どこでHTMLを作るか」ではなく、「コンポーネント」という概念です。

*   **Next.js (RSC - React Server Components)**:
    *   **思想**: ページを「サーバーで描画する部分」と「クライアントで動く部分」にパズルのように分割する。
    *   **メリット**: 必要なJSだけをユーザーに送るため、初期表示が爆速。SPAとMPA（マルチページ）のいいとこ取り。
    *   **デメリット**: 「これはServer？Client？」というメンタルモデルの切り替えが複雑。設計難易度が高い。
*   **Rails/Laravel (Classic MVC)**:
    *   **思想**: リクエストに対して、完成されたHTMLを「ドン」と返す。
    *   **メリット**: シンプルで堅牢。脳の負荷が低い。
    *   **デメリット**: 画面の一部を更新するだけでも、基本はページ全体の再取得になる（Hotwire等で改善はされているが、Reactの柔軟性には及ばない）。

## 2. Type Safety: End-to-End vs The Gap
開発体験（DX）において、最大の分水嶺がここです。

*   **Next.js (TypeScript)**:
    *   DBのスキーマから、バックエンドの処理、そしてフロントエンドのボタン一つに至るまで、**完全に「型」が繋がります**（tRPCやPrismaを使用した場合）。「DBのカラム名を変えたら、即座にボタンのコンポーネントでエラーが出る」という体験は、一度味わうと戻れません。
*   **Rails/PHP**:
    *   バックエンド（PHP/Ruby）とフロントエンド（JS）の間には、深くて暗い「型の溝」があります。API仕様書と実装がズレて、本番環境でだけJSエラーが出る…そんな悪夢は、このアーキテクチャの宿命です。

## 3. Data Mutation: Server Actions vs REST APIs
*   **Next.js (Server Actions)**:
    *   関数を呼び出す感覚でバックエンド処理を実行できます。「APIのエンドポイント設計」という概念すら希薄になります。
    *   **課題**: Next.jsアプリとバックエンドが密結合になるため、後から「スマホアプリ版も出したい」となった時に、APIの再設計が必要になることがあります。
*   **Rails/PHP (REST)**:
    *   「リソース」ベースの設計は、どんなクライアント（Web, iOS, Android）に対しても公平で汎用的です。

![Complexity Scale](/Engineerblog/window.svg)

## 4. Infrastructure: Edge vs Server
*   **Next.js**:
    *   Vercelなどのモダンなホスティング環境（Edge Network）との親和性が高い反面、特定のインフラへのロックインリスクがあります。「AWSのEC2にポンと置く」だけでは、Next.jsの真価（画像最適化やISR）を発揮しづらいのが現実です。
*   **Rails/PHP**:
    *   VPS、Docker、Heroku、レンタルサーバー。**人類が作ってきたあらゆるインフラで動きます**。この「枯れた安心感」は、B2Bの業務システムでは何物にも代えがたい価値です。

## 5. Complexity for "Simple" Sites
*   **Next.js**: 「HPを1枚作る」だけでも、Node.jsのバージョン管理、ビルド設定が必要です。単純なサイトには明らかに**オーバーキル**です。
*   **PHP**: `index.php` を書いてFTPで上げる。この「圧倒的な初速」に勝てる技術は、2025年現在でも存在しません。

# 警鐘：歴史は繰り返す

皮肉なことに、最新のNext.js App Routerを見ていると、**「React版のPHP」**に回帰しているように見えます。
SQLを直接コンポーネントの中で書き、Server Actionsでデータを送信する。これは、かつて私たちが「カオスになるから」と捨て去った、初期のPHPの姿に似ていませんか？

「流行っているから」という理由だけで採用するのは危険です（カーゴ・カルト）。
Reactエコシステムは強力ですが、それを維持するためのコスト（依存ライブラリの更新、破壊的変更への追従）は、思った以上に高いのです。

![Conclusion](/Engineerblog/vercel.svg)

# 結論：適材適所という「当たり前」に立ち返る

では、どう選ぶべきか？

1.  **Next.jsを選ぶべき時**: 
    *   ユーザー体験（UX）が競争力の源泉になる**B2Cサービス**。
    *   ダッシュボードなど、インタラクティブ性が極めて高いCMSやSaaS。
2.  **Rails/Laravelを選ぶべき時**: 
    *   「機能（Logic）」が主役の**複雑な業務システム**。
    *   少人数のチームで、MVP（実用最小限の製品）を**爆速でリリース**したい時。
    *   APIとして、モバイルアプリなど複数のクライアントに機能を提供する場合。

「銀の弾丸」はありません。
両方の特性を理解し、プロジェクトの性質に合わせて「武器」を持ち替えることができるエンジニアこそが、2025年に最も求められている人材です。
Next.jsの華やかさに目を奪われすぎず、Rails/PHPの堅実さも見直す。そんなバランス感覚を持って、次のプロジェクトに向き合ってください。
